# 基础技术框架

配合《核心玩法循环.md》，这份文档回答几个开发上必须先定下来的问题：场景怎么分、哪些逻辑在服务端哪些在客户端、数据怎么存、资源怎么加载才不卡。目标是给团队一个能直接开工的骨架，细节可以后续迭代。

---

## 1. 整体场景架构：家园 Place + 对局 Place

Roblox 支持一个游戏由多个 Place（场景文件）组成，玩家在 Place 之间用 `TeleportService` 传送。这套结构天然适合我们的"局外家园 + 局内对局"双循环，强烈建议**从第一天就分成两个 Place**，而不是在一个场景里用 Model 隐藏/显示来模拟：

```
游戏（Experience）
├── Place A：家园大厅（HomePlace）—— 常驻、持久化展示
│   每个玩家的 home、展示柜、置物架，以及"开一局"的入口
│
└── Place B：对局场景（RoundPlace）—— 每局一个独立服务器实例
    1人/2人/4人局共用同一份地图逻辑，按人数生成不同容量的房间
```

**为什么要分两个 Place，而不是一个场景走完：**

| 原因 | 说明 |
|---|---|
| 资源隔离 | 家园只需要加载装饰/展示类资源，对局只需要加载垃圾场地图和掉落物资源，互不拖累加载时间 |
| 服务器隔离 | 每局是独立的服务器实例（Reserved Server），一局崩了/卡了不影响其他玩家的家园体验 |
| 逻辑隔离 | 局内的翻找、掉落、结算是重逻辑，家园的展示是轻逻辑，分开写不容易互相污染状态 |
| 后续扩展 | 以后加新玩法模式，直接加新 Place，不用改家园代码 |

**分流方式**：家园里放一个"开局"交互点，玩家选 1/2/4 人模式后，服务端用 `TeleportService:ReserveServer()` 预留一个对局服务器实例，1 人局直接 `TeleportAsync`，2/4 人局用 `TeleportPartyAsync` 把整队一起传过去（这样才能保证组队的几个人落在同一个对局服务器里）。局结束后，服务端把玩家 `TeleportAsync` 传回家园 Place。

### 1.1 家园内每人一块地：Plot 分配机制

家园 Place 是所有在线玩家共享的一个服务器实例，但每个玩家在里面有**属于自己的一小块地**——自己的出生点 + 自己的展示柜。这块地不是按玩家账号在场景里各建一份（那样场景内容会随玩家数线性增长，无法维护），而是走**固定数量的物理地块槽位（Plot）+ 运行时动态分配归属**的模式：

```
家园场景（Studio 里固定摆好，不随玩家变化）
├── Plots/
│   ├── Plot_1（出生点 + 展示柜）
│   ├── Plot_2（出生点 + 展示柜）
│   ├── ...
│   └── Plot_N（出生点 + 展示柜）
└── RoundEntrances/（共享的开局传送点，所有玩家从各自的 Plot 走出来后一起去这里集合）
```

**分配逻辑（服务端）：**

```
玩家加入家园服务器（PlayerAdded）
    → 服务端从 Plot 池里找一个当前空闲（Occupied = false）的槎位
    → 标记该 Plot 为已占用，记录 assignedPlots[userId] = plotIndex（运行时内存表，不用持久化）
    → 把玩家角色传送/MoveTo 到该 Plot 的出生点
    → 加载该玩家的持久化数据，把 UnlockedItems 对应的物品模型 Clone 到这个 Plot 的展示柜上

玩家离开家园服务器（PlayerRemoving）
    → 释放该 Plot（Occupied = false），清空展示柜上 Clone 出来的物品模型
    → 这个 Plot 可以立刻分配给下一个新加入的玩家
```

几个关键点：

- **Plot 的物理位置是固定的**，在 Studio 里提前摆好一批（比如 8 个），不会因为换玩家而移动或重新生成。变化的只是"这个 Plot 现在归谁"这个逻辑归属，这个归属关系存在服务端运行时内存里，不用写 DataStore（断线重进/换服务器本来就应该重新分配，不需要"记住我上次用的是哪个 Plot"）。
- **展示柜渲染的目标位置变成动态的**：原来是"固定的共享柜子"，现在是"服务端先查这个玩家当前分配到哪个 Plot，再往那个 Plot 下的柜子 Clone 物品"。玩家的持久化数据结构（`UnlockedItems`）完全不用变，只是渲染时多了一步查 Plot 归属表。
- **Plot 数量是有限资源**：如果同时在线人数超过 Plot 数量，需要有兜底策略（比如排队等待空位，或者临时不分配专属 Plot 只给基础功能）。第一版先按预期同服人数摆够数量的 Plot，够用即可，不用一开始就做复杂的排队系统。
- **默认出生流程**：Roblox 自带的 `SpawnLocation` 只能随机选一个启用的出生点，没法按玩家分配到指定位置，所以实际做法是：保留一个共享的中立出生点（玩家角色刚加载出来的落脚点），角色生成后由服务端脚本立即把玩家 `MoveTo` 或传送到分配到的 Plot 出生点。这一步在 `PlayerAdded`/`CharacterAdded` 里做，不依赖 Roblox 默认的多 SpawnLocation 机制。

## 2. 服务端 vs 客户端：职责怎么分

**基本原则（不能违反）：服务端是唯一权威，客户端只负责"我想干什么"和"让我看到什么"。** 这不是洁癖，是因为 Roblox 的客户端本质是完全不可信的——外挂可以对任何 RemoteEvent 发任何参数。翻找/掉落/金钱这些直接影响体验公平性和经济系统的东西，**必须**在服务端算。

| 内容 | 服务端 | 客户端 | 备注 |
|---|---|---|---|
| 挖掘/翻找的结果判定 | ✅ 判定掉落物 | 只发"我在这个点翻找"的意图 | 客户端不能自己决定翻出了什么 |
| 掉落物稀有度/概率 | ✅ 服务端掉落表 | — | 掉落表数据可以放共享模块，但"抽"这个动作必须在服务端执行 |
| 局内金钱、能力提升 | ✅ 服务端计算和存储 | 只展示数值 | 参考"直接修改货币"是 Roblox 最常见的被刷漏洞 |
| 稀有物首次解锁判定 | ✅ 服务端判定并写入持久化数据 | 只展示"解锁"特效 | 解锁是要写永久数据的动作，必须服务端把关 |
| 翻找动作的手感（挥动动画、粒子、音效） | — | ✅ 客户端本地播放 | 不需要等服务端确认才能有反馈，减少操作延迟感 |
| 摄像机、UI、输入 | — | ✅ 全部客户端 | |
| 家园展示柜的物品渲染 | 服务端提供"这个玩家解锁了哪些物品"的数据 | ✅ 客户端负责把对应模型摆到柜子里 | 展示是纯表现层，不影响数值公平性 |

一个具体例子（翻找的完整流程）：

```
客户端：玩家点击某个垃圾堆 → 立刻本地播放挖掘动画（不等服务端，保证手感）
       → 发送 RemoteEvent "RequestDig"，带上垃圾堆的实例引用

服务端：收到 RequestDig
       → 校验：这个垃圾堆存在吗？玩家距离够近吗？冷却时间到了吗？
       → 通过校验后，服务端按掉落表抽一个结果
       → 更新服务端内存里这局的玩家状态（如果是稀有物，判断是否首次获得）
       → 通过 RemoteEvent 把结果广播回客户端

客户端：收到结果 → 播放对应的掉落特效/UI提示
```

客户端的"立刻播放动画"和服务端的"判定结果"是分开的两件事，这样既保证手感不延迟，又保证结果不可被伪造。

### 2.1 基础垃圾拾取：高频操作的专门方案

长按吸取几百个基础垃圾 Part 这个操作，跟上面"翻找稀有物"不是同一类问题——它不需要抽奖判定，但**换钱**这个动作本身是硬红线，加上高频、多人同时抢同一个 Part，需要单独设计。

**客户端预测 + 服务端裁决：**

```
客户端：长按检测范围内的垃圾 Part
       → 立刻本地播放"飞向背包"的视觉效果（不等服务端，保证手感跟手）
       → 每 0.1~0.2 秒把当前命中的 Part 列表打包发一次（节流，不逐个发）

服务端：收到一批 Part 请求
       → 逐个检查：这个 Part 还存在吗？没被其他玩家抢先标记吗？距离合理吗？
       → 通过：立刻标记/回收该 Part（防止同一帧内被别的玩家的请求重复通过），记入该玩家本局背包内存
       → 不通过：回复"拒绝"，客户端撤销本地那个虚假的飞入效果

回收站：一次性把背包里的垃圾数量在服务端汇总换算成金币，清空背包
       → 数量和金额必须服务端算总数，绝不能信客户端报的数字
```

**几个具体的优化点：**

| 优化点 | 原因 |
|---|---|
| Part 一旦被某请求命中就立刻标记/回收，而不是简单判断"存在与否" | 2/4 人局里两个玩家可能同帧碰到同一个 Part，简单的存在性判断会导致双计数，凑出无中生有的额外垃圾 |
| 客户端节流打包发送，不要每碰到一个就发一次 Remote | 几百个 Part 高频命中时，逐个发送会产生大量不必要的网络请求 |
| 垃圾 Part 用 `CollectionService` 打 Tag（如 "Trash"）统一标记，而不是按名字/路径查找 | 数量一大，`FindFirstChild` 类查找性能会线性下降，Tag 方式查找更稳定 |
| 拾取/回收后走对象池，不要 `Destroy()` | 垃圾场会反复刷新同类型 Part，池化复用比反复 New/Destroy 开销小很多，参考第 4.2 节 |
| 服务端只做"计数"，不做"判定" | 基础垃圾不需要跑掉落表抽奖，服务端校验只需确认"Part 有效 + 没被抢 + 距离合理"，然后按 Part 自身的价值属性记账，逻辑比稀有物简单，不用套用抽奖那一套增加不必要的复杂度 |
| 背包数据留在局内内存，不接 DataStore | 回收换的钱本来就不带出局，跟局内金钱的存储方式一致，不涉及持久化 |

## 3. 数据存储：分清"局内临时"和"局外永久"

这是整个技术框架里最容易踩坑的一块。按照核心玩法，我们数据其实分两层，**存储方式完全不同**：

### 3.1 局内数据（临时，不落库）

单局内的钱、买的能力提升，局结束就清空，**完全没必要写数据库**，直接放在对局服务器的内存里（一个 Lua table）就行。这样做的好处是零延迟、零 DataStore 配额消耗。

```lua
-- 对局服务器内存里维护，不经过 DataStore
local roundSessions = {}
-- roundSessions[player.UserId] = { coins = 0, upgrades = {digSpeed = 1, ...} }
```

局结束（时间到/退出）时，只需要把"这局里首次获得的稀有物列表"这一小段数据，作为结算结果处理，其他都可以直接丢弃。

### 3.2 局外数据（永久，需要 DataStore）

需要长期保存的只有：**哪些稀有物被解锁过**（图鉴/展示柜数据）、以及未来可能加的家园布局、玩家等级等成长数据。这部分才需要走 DataStore，且必须做到不丢数据。

推荐用社区标准的 **ProfileService**（或其继任者 ProfileStore）来管理，而不是自己裸写 `GetAsync`/`SetAsync`——它自带会话锁（防止同一存档在两台服务器同时写导致的数据错乱）、自动重试、服务器关闭前保存等机制，这些是新手最容易漏掉的地方。数据结构草案：

```lua
-- 玩家永久档案（DataStore，走 ProfileService）
{
    UnlockedItems = { -- 图鉴/展示柜解锁记录
        ["item_rare_001"] = true,
        ["item_rare_017"] = true,
    },
    HomeLayout = { -- 展示柜/置物架的摆放（如果允许玩家自定义摆放顺序）
        shelf1 = "item_rare_001",
        shelf2 = "item_rare_017",
    },
    Stats = {
        RoundsPlayed = 0,
        TotalItemsFound = 0,
    },
    Version = 1, -- 预留字段，未来数据结构变了方便迁移
}
```

**几条硬性规则（新手最容易翻车的地方）：**
- 所有 DataStore 读写必须包在 `pcall` 里，失败要重试，不能静默丢失。
- 不要频繁保存：自动保存间隔建议 60-120 秒，不要做成"每次翻找都存一次"，DataStore 有请求频率限制，量一大直接触发限流导致存档失败。
- 必须在 `game:BindToClose()` 里保存，否则服务器关闭瞬间在线的玩家会丢当前进度。
- 解锁稀有物这种关键动作，建议**立即触发一次保存**（不用等自动保存周期），避免玩家刚解锁就掉线导致白解锁。

### 3.3 局内结果怎么"带回"家园

局结束时的流程：

```
对局服务器：结算 → 把本局解锁的稀有物 ID 列表，直接写入玩家的持久化档案（DataStore）
           → 保存成功后，TeleportAsync 把玩家传回家园 Place

家园服务器：玩家进入 → 服务端分配一个空闲 Plot（见 1.1 节，这一步跟持久化数据无关，纯运行时逻辑）
           → 重新加载该玩家的持久化档案
           → 根据 UnlockedItems 数据，把对应模型摆到分配到的 Plot 的展示柜上
```

关键点是：**解锁数据的写入发生在对局服务器结算的那一刻，不是家园服务器**。这样即使玩家传送过程中断线，解锁记录已经落库了，不会丢。Plot 分配是每次进家园服务器都重新做一次的运行时行为，和"哪些物品解锁了"这个持久化事实完全独立——换一台家园服务器、断线重进，物品数据不受影响，只是可能分到不同的 Plot 编号。

## 4. 资源加载与性能：怎么做到不卡

### 4.1 家园 Place：按需加载展示物，不要全量常驻

家园是所有玩家共享访问的持久场景，如果每个人的展示柜都把"所有可能的稀有物模型"提前摆好只是隐藏起来，场景的 Part 数量会随着物品库扩张线性爆炸。正确做法：

- 展示柜/置物架本身（静态建筑部分）是场景自带的，一直存在，且数量固定（等于 Plot 数量），不随玩家数变化。
- 具体摆在柜子里的**物品模型**，只在"这个 Plot 当前分配到的玩家解锁了该物品"时才从 `ReplicatedStorage` 里 Clone 一份放上去，而不是把全物品库都摆进场景。
- 玩家离开 Plot（下线/传送去对局）时，要把该 Plot 展示柜上 Clone 出来的物品模型清空，避免下一个分配到这个 Plot 的玩家看到上一位的物品残留。
- 所有物品的模型原件统一放在 `ReplicatedStorage/ItemModels/` 下，只加载一次，按需 Clone。

### 4.2 对局 Place：垃圾堆和掉落物走对象池，不要现场 New

翻找是高频动作，如果每次翻找都 `Instance.new()` 生成掉落物、卖出/收集后 `Destroy()`，短时间内大量创建销毁会造成明显的性能抖动。改用对象池：

```lua
-- 掉落物对象池：翻找出来的物品从池里取，卖掉/收进背包后归还池子而不是销毁
local ItemPool = {}
function ItemPool.Get(itemType)
    -- 有空闲实例就复用，没有才 Clone 新的
end
function ItemPool.Return(instance)
    instance.Parent = nil -- 移出场景但不销毁，留给下次复用
end
```

垃圾堆本身如果数量固定（比如每个场景 20-30 个堆），可以直接常驻不用池化，重点池化的是"翻找出来的高频小物件"。

### 4.3 场景规模大的话，开 StreamingEnabled

如果对局地图或家园大厅面积比较大（尤其家园是多人共享的开放大厅，视野内其他玩家的家会占资源），建议打开 `Workspace.StreamingEnabled`，让引擎按玩家距离自动加载/卸载远处的场景内容，减少同屏渲染压力。这对手机端玩家（Roblox 上占比很高）尤其重要。

### 4.4 加载画面阶段预加载资源

进入对局前，用 `ContentProvider:PreloadAsync()` 把这局地图需要的物品模型、贴图预先加载好，避免玩家进场景后看到东西一个个"弹出来"（pop-in），这个可以配合传送时的 Loading UI 一起做。

## 5. 网络通信：RemoteEvent 怎么组织

建议在 `ReplicatedStorage/Remotes/` 下按功能分类放 RemoteEvent/RemoteFunction，命名清晰、集中管理，不要散落在各个脚本里现场 `Instance.new("RemoteEvent")`：

```
ReplicatedStorage/
└── Remotes/
    ├── Round/
    │   ├── RequestDig         (客户端→服务端：翻找意图)
    │   ├── DigResult           (服务端→客户端：翻找结果)
    │   ├── SellItem            (客户端→服务端：卖出物品)
    │   └── PurchaseUpgrade     (客户端→服务端：局内能力提升购买)
    └── Home/
        └── GetHomeData         (客户端→服务端：请求展示柜数据，服务端按当前 Plot 归属返回)
```

每一个服务端接收的 Remote，落地代码里都要有三件事（缺一不可，这也是 Roblox 游戏被刷的最主要原因）：
1. **类型校验**：参数类型对不对，是不是恶意构造的值。
2. **状态校验**：玩家距离够不够、冷却到了没、这个东西是否真的存在。
3. **频率限制**：防止脚本疯狂调用同一个 Remote 刷东西。

如果后期翻找这类高频操作对网络性能有明显压力，可以考虑换用社区的高性能网络库（比如 Zap），但项目初期用原生 RemoteEvent + 规范校验完全够用，不必过早引入。

## 6. 建议的项目文件夹结构

```
ServerScriptService/
├── Round/
│   ├── RoundManager.lua        -- 对局生命周期：开始/结算/传送回家园
│   ├── DigHandler.lua          -- 处理翻找请求，服务端权威判定
│   └── LootRoller.lua          -- 掉落表抽取逻辑
├── Home/
│   ├── PlotManager.lua         -- Plot 池管理：分配/释放、玩家↔Plot 归属表（运行时内存，不落库）
│   └── HomeDataService.lua     -- 家园展示数据的读取与推送，按 Plot 归属摆放物品模型
└── Data/
    └── ProfileManager.lua      -- 基于 ProfileService 的持久化数据管理

ReplicatedStorage/
├── Remotes/                    -- 见第 5 节
├── Modules/
│   ├── LootTable.lua           -- 掉落表配置（数据驱动，方便策划调数值不动代码）
│   └── ItemDefinitions.lua     -- 所有物品的定义（稀有度/售价/模型引用等）
└── ItemModels/                 -- 所有物品的模型原件，按需 Clone

StarterPlayerScripts/
├── DigController.lua           -- 本地翻找手感（动画/粒子），发起 RequestDig
└── UIController.lua            -- 背包/图鉴/卖出等 UI 逻辑
```

`LootTable` 和 `ItemDefinitions` 单独拆成数据模块是故意的：掉落概率、物品稀有度、售价这些是策划会反复调的数值，跟逻辑代码分开放，改数值不需要碰逻辑代码，也方便后续做数值平衡。

## 7. 版本管理：Studio 官方 Script Sync（已采用，不用 Rojo）

项目已确定使用 Roblox Studio 官方的 **Script Sync** 功能做本地文件同步，配合 Git 做版本管理，不引入 Rojo。

### 7.1 Script Sync 是什么，跟 Rojo 的区别

- Script Sync 只同步 `Script`/`LocalScript`/`ModuleScript`/`Folder` 这四种实例，双向同步到本地 `.luau` 文件——Studio 里改代码会同步到本地文件，本地文件改了也会同步回 Studio。
- 定位是"Studio 为准"：正常在 Studio 里搭建场景、写代码，Script Sync 只是把脚本实例镜像成本地文件方便用外部编辑器/linter/git diff，**不接管场景搭建**。这点跟"文件系统为准"的 Rojo 不同，Rojo 需要把整个项目结构写在本地再推到 Studio，容易覆盖 Studio 里手动搭的场景。我们的场景（Home 地块、RoundEntrances 等）都是手动搭的，Script Sync 这种"Studio 为准"的模式更贴合现在的工作习惯。
- **硬限制**：一个被同步的 Folder 里如果混了脚本和非脚本实例（比如 Part、Model），非脚本的部分会被忽略。所以 `Remotes/`（RemoteEvent 实例）、`ItemModels/`（模型原件）这类非脚本内容**不会**、也不应该被同步进本地文件，只存在于 Studio 里，靠《架构与Remote登记.md》和《开发记录.md》人工记录现状。
- **文件名后缀决定同步生成的实例类型，改名时不能随便动这个后缀**（吃过一次亏，见开发记录 2026-07-07）：
  | 文件名后缀 | 生成的实例 | 说明 |
  |---|---|---|
  | `.server.luau` | `Script`（RunContext: Server） | 服务端脚本，放 ServerScriptService 会自动运行 |
  | `.client.luau` | `LocalScript` | 客户端脚本 |
  | `.legacy.luau` | `Script`（RunContext: Legacy） | 旧式脚本，行为等同放对容器里自动运行 |
  | 纯 `.luau`（无以上后缀） | `ModuleScript` | **不会自动运行**，必须被其他脚本 `require()` 才执行 |

  把一个原本是 `.server.luau`/`.legacy.luau` 的文件手动改名去掉后缀，会让 Script Sync 把它同步成 ModuleScript——代码还在，但因为没人 `require` 它，会安静地什么都不做，也不会报错，很容易误以为是逻辑问题去排查半天。

### 7.2 目录映射：按 Place 分两个顶层目录

家园 Place 和对局 Place 是两个独立的 DataModel，各自在 Studio 里单独配置 Script Sync 的同步目录，对应仓库里的两个顶层目录。**实测发现**：对某个实例执行 "Sync to..." 选择本地目录后，Script Sync 会在目标目录下自动多生成一层跟该实例同名的子文件夹（比如对 `ServerScriptService` 选了 `home/ServerScriptService` 作为同步目录，实际文件会落在 `home/ServerScriptService/ServerScriptService/...` 下），这是它的固定行为，不是配置错误，以下路径已按实测结果修正：

```
jianlaji/
├── home/                                                              -- 家园 Place（placeName: Place1）
│   ├── ServerScriptService/ServerScriptService/Home/                -- RoundEntranceService.server.luau（V1.1/V1.1.1）、PlotManager.server.luau、HomeDataService.server.luau
│   ├── ReplicatedStorage/Modules/Modules/                            -- 家园侧需要的数据模块（纯 .luau，ModuleScript）
│   └── StarterPlayer/StarterPlayerScripts/StarterPlayerScripts/     -- EntranceLeaveController.client.luau（V1.1.1）
└── battle/                                                          -- 对局 Place（placeName: battle）
    ├── ServerScriptService/ServerScriptService/Round/               -- RoundManager.server.luau、DigHandler.server.luau、LootRoller.server.luau
    ├── ServerScriptService/ServerScriptService/Data/                 -- ProfileManager.server.luau
    ├── ReplicatedStorage/Modules/Modules/                             -- LootTable.luau、ItemDefinitions.luau（纯 .luau，ModuleScript）
    └── StarterPlayer/StarterPlayerScripts/StarterPlayerScripts/     -- DigController.client.luau、UIController.client.luau
```

**踩过两次坑，最终确认的规律**：`StarterPlayerScripts` 在 Roblox 里不是独立顶层 Service，而是嵌套在 `StarterPlayer` 服务下面（真实路径 `StarterPlayer.StarterPlayerScripts`）。对它执行 Sync to 时：
1. 本地目标目录要选 `home/StarterPlayer/StarterPlayerScripts`（对局 Place 选 `battle/StarterPlayer/StarterPlayerScripts`），如果直接选 `home/StarterPlayerScripts` 会报"所选路径与层次结构 'StarterPlayer.StarterPlayerScripts' 不匹配"，Studio 还可能借机弹冲突框把本地文件覆盖删除（第一次踩坑，见开发记录 2026-07-07）。
2. 跟 `ServerScriptService`/`ReplicatedStorage` 一样，Script Sync 还是会在这个目标目录下**再自动生成一层同名子文件夹**，所以脚本真正落地的路径是 `home/StarterPlayer/StarterPlayerScripts/StarterPlayerScripts/`，比目标目录多一层（第二次踩坑，文件放少了一层导致 Studio 没识别到）。

后缀规则见上方 7.1 节的表格：服务端脚本用 `.server.luau`，客户端脚本用 `.client.luau`，数据模块用纯 `.luau`（ModuleScript）。

**已知限制（两个 Place 独立 DataModel 带来的）：**两边如果需要用到同一份数据模块（比如 `ItemDefinitions.luau` 家园展示和对局翻找都要读物品定义），没有自动共享机制，只能各自维护一份，改动时需要手动同步内容保持一致，容易漏改。目前物品数据规模不大，先手动维护，规模变大后再考虑要不要抽成 Wally 包之类的方案。

### 7.3 操作步骤（在 Studio 里配置）

1. 打开对应的 Place（先用 `get_place_info` 确认连的是家园还是对局，避免配错）。
2. 在 Explorer 里右键要同步的文件夹（比如 `ServerScriptService`），选择 **"Sync to..."**。
3. 选择本地目录，指向仓库里对应的路径（家园 Place 选 `home/ServerScriptService`，对局 Place 选 `battle/ServerScriptService`）。**注意**：Script Sync 会在选定目录下再自动建一层同名子文件夹，实际文件会出现在 `home/ServerScriptService/ServerScriptService/` 下，这是正常现象。
4. `ReplicatedStorage` 只对 `Modules` 子文件夹做 Sync to，不要对整个 `ReplicatedStorage` 做（会把 `Remotes`/`ItemModels` 也带进去，非脚本内容会被忽略且容易造成困惑）。
5. **`StarterPlayerScripts` 要单独注意**：右键 Explorer 里的 `StarterPlayerScripts` 选 Sync to 时，本地目标目录要选 `home/StarterPlayer/StarterPlayerScripts`（对局 Place 选 `battle/StarterPlayer/StarterPlayerScripts`），不要选成 `home/StarterPlayerScripts`。绑定后脚本实际会落地在再多一层的 `home/StarterPlayer/StarterPlayerScripts/StarterPlayerScripts/` 下（跟第3步的自动多一层规律一致）。原因见上方 7.2 节的说明。
6. 两个 Place 分别重复以上步骤，各自绑定到 `home/` 和 `battle/` 下对应的子目录。

配置完成后，脚本改动就能被 git 追踪和 diff；场景搭建、Tag/Attribute、RemoteEvent 实例等非脚本内容仍然只存在于 Studio 里，改动后要记得写进《开发记录.md》。

## 小结

这套框架的核心思路就四条：
1. **家园和对局分两个 Place**，各自只加载自己需要的东西。
2. **家园里每人一块 Plot**，物理位置固定、运行时动态分配归属，不随玩家数增长场景内容。
3. **局内数据全在内存，局外数据才落 DataStore**，中间只有"结算时把解锁记录写库"这一个动作连接两者；Plot 归属本身也是运行时数据，和持久化的解锁记录是两件独立的事。
4. **服务端管结果和权威数据，客户端管手感和展示**，任何涉及钱/概率/解锁的判定绝不能交给客户端。

下一步如果需要，可以基于这个框架开始搭 Studio 里的实际目录结构和最基础的翻找-掉落-结算跑通链路。
