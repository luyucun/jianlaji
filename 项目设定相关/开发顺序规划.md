# 开发顺序规划

配合《核心玩法循环.md》和《基础技术框架.md》。这份文档回答"接下来按什么顺序开发"——策划案按这个顺序分阶段输出，技术实现照着同一顺序接，两边不脱节。

## 判断依据

1. 这个项目最大的不确定性是"翻找+随机掉落"的核心爽感是否好玩，应该最先做出可验证的最小闭环，而不是先把家园/持久化/多人这些外围机制搭完整。
2. 局内数据不落库、局外数据才落库——这个分层决定了"持久化"可以晚接入，不影响早期验证核心玩法。
3. Roblox 平台限制：新增 Place（对局 Place）需要先发布游戏。这个约束会影响"核心循环该在哪个场景先验证"的选择，留到对应阶段再决定技术细节，不影响策划案的输出顺序。
4. 多人（2/4人）在同一个 Part 上的并发风险（双计数）比单人复杂得多，应该先在单人场景验证判定逻辑正确，再扩展到多人。
5. 《架构与Remote登记.md》已有 3 个"设计中"的 Remote（RequestDig/RequestPickup/RequestSell），正好对应第一批要落地的核心循环，从这里切入最顺。

---

## 推荐开发顺序（分 10 个阶段）

### 阶段 1：物品与掉落数据地基
- 策划产出：完整物品清单（普通垃圾 / 可出售物 / 稀有物三层），每层的掉落概率、售价；稀有物与"首次解锁"的对应关系。
- 技术产出：`ReplicatedStorage/Modules/ItemDefinitions.lua`、`LootTable.lua`（数据驱动，占位数值即可，后续改数值不碰逻辑代码）。
- 依据：所有后续机制（翻找判定、变现、解锁）都要读这份数据，越早定下结构越不用返工。

### 阶段 2：单人局核心循环最小可玩版（全项目风险最高的一步，优先验证）
- 策划产出：翻找的具体交互规则（点击/长按判定细节，对应参考图的手感）、单局时长/结束条件、局内能力提升的维度定义（翻找速度/范围/移动等，先给 1~2 个维度即可，不用一次列全）。
- 技术产出：落地《架构与Remote登记.md》里已登记的 `RequestDig`/`RequestPickup`/`RequestSell`，服务端权威判定（`DigHandler.lua`/`LootRoller.lua`），客户端手感（`DigController.lua`）。范围先限定单人，不处理多人并发。
- 技术决策点（到这一步再定，不影响现在的策划案输出）：Roblox 要求先发布游戏才能新建对局 Place，这里可以选择"先发布游戏正式建对局 Place"或"在当前 Place 里开一块临时场景验证手感、验证完再搬进正式对局 Place"，两种都不影响策划案内容。

### 阶段 3：局内经济与成长打磨
- 策划产出：能力提升的定价表和加成数值曲线。
- 技术产出：`PurchaseUpgrade` Remote（需先补进 Remote 登记表），局内内存数据结构定稿。

### 阶段 4：结算与稀有物解锁判定
- 策划产出：稀有物解锁规则细则（首次获得如何判定、单局是否有解锁上限）。
- 技术产出：`RoundManager` 结算逻辑，"首次获得"写入接口（先用内存 Mock，不接 DataStore，验证规则本身对不对）。

### 阶段 5：局外持久化接入
- 策划产出：基本不需要策划输出，确认 `UnlockedItems` 数据结构即可。
- 技术产出：接入 ProfileService，`BindToClose` 保存、解锁即时保存规则落地。

### 阶段 6：家园循环——Plot 归属与展示柜渲染
- 策划产出：展示柜/置物架的呈现规则（一个稀有物对应哪个展示位，是否允许玩家自定义摆放顺序）。
- 技术产出：给 `Home01`~`Home15` 补上 Tag/Attribute（或按名字遍历，两种都可，到时候定），`PlotManager.lua` 分配逻辑，`HomeDataService.lua` 渲染逻辑；目前 Home 地块下还没有展示柜模型，需要先补上。

### 阶段 7：多人局（2人/4人）扩展
- 策划产出：组队规则（是否要求好友、匹配方式）。
- 技术产出：`TeleportPartyAsync` 分流，重点验证 2/4 人同时抢同一垃圾 Part 的并发标记方案（阶段 2 已验证单人逻辑正确后再扩展，降低调试复杂度）。

### 阶段 8：双 Place 正式分离与传送打磨
- 若阶段 2 用了临时方案验证手感，这一步把对局内容正式搬进独立的对局 Place，接入正式 `TeleportService` 流程、Loading UI、`PreloadAsync`。

### 阶段 9：性能与体验打磨
- 对象池化（`ItemPool`）、`StreamingEnabled`、UI 精修、占位方块替换成正式美术。

### 阶段 10：数值平衡与内容扩充
- 更多物品种类、更多稀有度层级、更多能力提升维度，正式进入长期运营调优节奏。

---

## 说明

- 阶段 2 是全项目唯一的高风险验证点：建议先只做最简单的版本（几个占位方块代表垃圾堆），跑通"翻找出乐子"这件事之后，再回头补齐阶段 3 及以后的完整性，不要在核心循环没验证好之前投入太多在家园/持久化/多人上。
- 每个阶段落地代码时仍要遵守 README 的硬规则：新增/改 Remote 先更新登记表、每个任务对应一次干净提交。
